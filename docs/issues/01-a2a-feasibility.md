# 飞书插件 A2A 可行性分析

## 概述

A2A (Agent-to-Agent) 协议允许多个 DMR Agent 在同一个飞书群聊中协作完成任务。本文分析在飞书平台上实现 A2A 的技术可行性、限制和推荐方案。

## 什么是飞书 A2A

与 Matrix A2A 类似，飞书 A2A 允许：

- 单个插件管理多个飞书 Bot（企业应用）
- Bot 之间通过飞书消息传递任务和结果
- 用户视角：多个 Bot 在群聊中协作对话

```
用户: @开发助手 帮我实现登录功能

@开发助手: 好的，我来编写代码...
@开发助手: 代码已完成，需要安全审查。@安全助手 请审查

@安全助手: 收到，开始安全审查...
@安全助手: 发现2个问题：1. SQL注入风险 2. 密码明文存储

@开发助手: 感谢反馈，我来修复...
@开发助手: 已修复，请再次审查

@安全助手: ✅ 审查通过

@开发助手: 功能开发完成并通过审查！
```

## 技术可行性

### 核心能力矩阵

| 能力 | 支持度 | 说明 |
|------|--------|------|
| **多 Bot 管理** | ✅ 完全支持 | 飞书插件已支持多租户模式 |
| **消息接收** | ⚠️ 有条件支持 | 需要申请 "获取群聊所有消息" 权限 |
| **消息发送** | ✅ 完全支持 | 标准消息发送 API |
| **消息格式** | ⚠️ 有限支持 | 富文本/卡片格式，非纯文本 |
| **实时推送** | ✅ 支持 | WebSocket / Event Subscription |
| **@提及** | ✅ 支持 | 飞书消息可以@用户和 Bot |

### 关键限制

#### 1. 消息可见性限制（核心限制）

飞书 Bot 在群聊中的消息可见性取决于应用权限配置：

```
飞书权限模式 A（默认）: "仅接收@机器人的消息"
├── Bot A 发送消息
├── Bot B 无法看到（除非被@）
└── ❌ A2A 无法工作

飞书权限模式 B（需申请）: "获取群聊中所有消息"
├── Bot A 发送消息  
├── Bot B 可以看到所有消息
└── ✅ A2A 可以工作
```

**权限申请要求**:
- 需要在飞书开放平台申请 "获取群聊中所有消息" 能力
- 需要企业管理员审批
- 公网应用较难获批

#### 2. 消息格式限制

飞书消息格式 vs Matrix：

| 特性 | Matrix | 飞书 |
|------|--------|------|
| 消息类型 | 纯文本 JSON | 富文本/卡片/文本 |
| 自定义字段 | 任意 JSON | 受限，有 schema 约束 |
| 长度限制 | 约 64KB | 约 30KB（卡片） |
| 二进制数据 | 可 base64 | 需要上传文件 |

## 与 Matrix A2A 对比

| 维度 | Matrix A2A | 飞书 A2A |
|------|------------|----------|
| 协议基础 | Matrix CS API | 飞书开放 API |
| 消息可见性 | 天然可见 | 需申请权限 |
| 消息格式 | JSON 纯文本 | 富文本/卡片 |
| 多 Bot 管理 | 单插件多账号 | 单插件多应用 |
| 权限门槛 | 低 | 高 |
| 部署难度 | 简单 | 复杂 |
| 适用场景 | 公网/私网 | 企业内网为主 |

## 可行性结论

### 结论：✅ 可行，但有条件

飞书 A2A **技术上可以实现**，但需要满足以下条件：

1. **权限条件**: 申请并获得 "获取群聊中所有消息" 权限
2. **部署条件**: 适合企业内部部署，公网应用较难
3. **实现条件**: 需要处理消息格式转换和长度限制

### 推荐度评估

| 场景 | 推荐度 | 说明 |
|------|--------|------|
| 企业内部飞书 | ⭐⭐⭐⭐ | 可以申请权限，推荐实现 |
| 公网 SaaS 产品 | ⭐⭐ | 权限难获批，不建议 |
| 个人开发者 | ⭐ | 几乎无法获批 |

## 替代方案：DMR Subagent

在飞书场景下，更推荐使用 DMR 内置的 `subagent` 工具实现 Agent 协作：

```
飞书 A2A:                    DMR Subagent:
Bot A ──► 飞书消息 ──► Bot B    Bot A ──► subagent 工具 ──► Agent B
   │                          │
   └── 依赖飞书权限              └── 完全在 DMR 内部
```

对比见 [02-a2a-vs-subagent.md](02-a2a-vs-subagent.md)
