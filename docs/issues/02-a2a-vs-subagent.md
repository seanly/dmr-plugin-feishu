# A2A vs Subagent：飞书场景下的选择

## 概述

在飞书场景下实现多 Agent 协作，有两种方案：

1. **飞书 A2A**: Bot 之间通过飞书消息协作
2. **DMR Subagent**: 使用 DMR 内置的 `subagent` 工具

本文对比两种方案，帮助选择适合的实施路径。

## 方案对比

### 架构对比

#### 方案 1：飞书 A2A

```
┌─────────────────────────────────────────────────────────────┐
│                     飞书群聊                                 │
│                                                              │
│  用户: @开发助手 帮我写代码并审查                              │
│                                                              │
│  @开发助手 Bot ───┐                                          │
│                   │                                          │
│                   ▼                                          │
│           飞书消息: "@安全助手 🔒A2A Task..."                   │
│                   │                                          │
│                   ▼                                          │
│  @安全助手 Bot ───┘                                          │
│  "收到，开始审查..."                                         │
│                                                              │
│  @安全助手 Bot ───┐                                          │
│                   │                                          │
│           飞书消息: "@开发助手 🔒A2A Result..."                 │
│                   │                                          │
│                   ▼                                          │
│  @开发助手 Bot ───┘                                          │
│  "审查完成，结果：..."                                        │
│                                                              │
└─────────────────────────────────────────────────────────────┘
         │
         │ 需要 "获取所有消息" 权限
         ▼
   飞书服务器
```

**特点**:
- Bot 之间通过飞书消息通信
- 用户可以看到完整的协作过程
- 依赖飞书权限

#### 方案 2：DMR Subagent

```
┌─────────────────────────────────────────────────────────────┐
│                     飞书群聊                                 │
│                                                              │
│  用户: @开发助手 帮我写代码并审查                              │
│                                                              │
│  @开发助手 Bot                                                │
│  "好的，我来处理..."                                          │
│       │                                                      │
│       ▼                                                      │
│  ┌───────────────────────────────────────────────────────┐  │
│  │              DMR 内部（用户不可见）                      │  │
│  │                                                        │  │
│  │  1. 开发 Agent 写代码                                   │  │
│  │       │                                                │  │
│  │       ▼                                                │  │
│  │  2. 调用 subagent 工具                                  │  │
│  │     subagent(name="security-reviewer", prompt="...")   │  │
│  │       │                                                │  │
│  │       ▼                                                │  │
│  │  3. 安全 Reviewer Agent 执行审查                        │  │
│  │       │                                                │  │
│  │       ▼                                                │  │
│  │  4. 返回审查结果                                        │  │
│  │                                                        │  │
│  └───────────────────────────────────────────────────────┘  │
│       │                                                      │
│       ▼                                                      │
│  汇总结果发送给用户                                           │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**特点**:
- Agent 协作在 DMR 内部完成
- 用户只能看到最终结果（或简要进度）
- 不依赖飞书特殊权限

### 详细对比表

| 维度 | 飞书 A2A | DMR Subagent |
|------|----------|--------------|
| **权限要求** | 需申请 "获取所有消息" 权限 | 无需特殊权限 |
| **部署难度** | 高（需审批） | 低（开箱即用） |
| **实现复杂度** | 高（消息格式、去重、防循环） | 低（DMR 内置） |
| **协作可见性** | 用户可见完整过程 | 用户仅见结果 |
| **延迟** | 高（需往返飞书服务器） | 低（内部调用） |
| **可靠性** | 依赖飞书服务 | 完全可控 |
| **调试难度** | 难（涉及外部消息） | 易（内部日志） |
| **扩展性** | 受限于飞书限制 | 灵活 |
| **适用场景** | 需用户参与决策的协作 | 自动化任务链 |

### 使用场景对比

#### 场景 1：代码审查流水线

**飞书 A2A 适用** ⭐⭐⭐⭐

```
用户: @开发助手 帮我实现功能并审查

开发 Bot: 代码已写好
安全 Bot: 发现安全问题，建议修改
开发 Bot: 已修改，请再次审查
安全 Bot: ✅ 通过
测试 Bot: 开始测试...
测试 Bot: ✅ 测试通过
开发 Bot: 功能已完成开发、审查和测试！
```

**原因**: 用户需要看到审查意见并参与修改决策

**DMR Subagent 适用** ⭐⭐⭐

```
用户: @开发助手 帮我实现功能

开发助手: 好的，我来处理...
[内部调用 subagent 完成代码生成、审查、测试]
开发助手: ✅ 已完成！结果如下：
        - 代码生成：完成
        - 安全审查：通过
        - 单元测试：通过
```

**原因**: 标准化的流水线，无需用户干预

#### 场景 2：多专家会诊

**飞书 A2A 适用** ⭐⭐⭐⭐⭐

```
用户: @架构助手 帮我设计系统

架构 Bot: 初步设计完成，需要安全评估
安全 Bot: 建议增加鉴权模块
性能 Bot: 建议增加缓存层
架构 Bot: 综合各位专家意见，最终设计如下：
        1. xxx
        2. xxx
```

**原因**: 用户需要看到各专家的独立意见

#### 场景 3：自动化报告生成

**DMR Subagent 适用** ⭐⭐⭐⭐⭐

```
用户: @数据助手 生成月度报告

数据助手: 正在处理...
[内部协调数据收集、分析、可视化多个 Agent]
数据助手: 报告已生成！
```

**原因**: 纯自动化流程，无需用户参与过程

## 混合方案

最佳实践可能是**混合使用**两种方案：

```
用户: @主控助手 完成复杂项目

主控 Bot: 收到，启动项目...
        
        [阶段 1：自动化流水线 - 使用 Subagent]
        ├── 代码生成 Agent
        ├── 自动测试 Agent  
        └── ✅ 自动化阶段完成
        
        [阶段 2：需要决策的环节 - 使用 A2A]
        ├── @安全专家 Bot 安全审查
        ├── @架构专家 Bot 架构审查
        └── 等待用户确认...

主控 Bot: 各位专家的审查意见如下：
        - 安全专家：建议 xxx
        - 架构专家：建议 xxx
        
        请用户确认是否继续？

用户: 确认，继续

主控 Bot: 继续执行...
        [继续 Subagent 自动化]
```

## 选择建议

### 选择飞书 A2A 如果：

- ✅ 企业内部飞书，可以申请到权限
- ✅ 用户需要看到并参与多 Agent 协作过程
- ✅ 各 Agent 代表不同角色/部门，需要独立发言
- ✅ 需要人工审批的协作流程

### 选择 DMR Subagent 如果：

- ✅ 公网部署，无法获取飞书特殊权限
- ✅ 协作过程对用户透明即可
- ✅ 追求低延迟和高可靠性
- ✅ 标准化、自动化的任务链
- ✅ 快速原型开发

### 推荐决策树

```
是否需要用户看到多 Agent 协作过程？
├── 是 ──► 是否企业内部飞书？
│           ├── 是 ──► 可以申请权限？
│           │           ├── 是 ──► ✅ 飞书 A2A
│           │           └── 否 ──► ⚠️ 考虑 Subagent + 进度反馈
│           └── 否 ──► ⚠️ 很难实现，考虑 Subagent
└── 否 ──► ✅ DMR Subagent（推荐）
```
